Design — Section 4: Gaze Interaction & View Swapping

How head tracking triggers hello, and how we switch between SceneKit and SpriteKit

Gaze → Character Mapping

Head Tracking Integration
Existing HeadTracker output:
  offset.x: -1.0 ... +1.0  (left to right)
  offset.y: -1.0 ... +1.0  (down to up)

Mapping to screen position:
  gazeX = screenWidth  * 0.5 + offset.x * screenWidth  * 0.5
  gazeY = screenHeight * 0.5 + offset.y * screenHeight * 0.5

Hit testing (per frame in SKScene.update):
  1. Convert gaze point to scene coordinates
  2. Find nearest CharacterNode within hitbox radius
     (hitbox = character size * 1.5 for comfortable targeting)
  3. If character found AND not already focused:
     → Start 300ms dwell timer
  4. If same character still focused after 300ms:
     → Trigger hello bubble fade-in (0.3s)
  5. If gaze leaves hitbox:
     → Cancel dwell timer
     → Fade out hello bubble (0.3s)

Performance: distance check is O(n) where n = session count.
With <20 sessions this is trivial in the frame callback.

SpriteKit Head Tracking Adapter

Reusing HeadTracker for SKScene
Current (SceneKit):
  SCNSceneRendererDelegate.renderer(_:updateAtTime:)
  → reads HeadTracker.latestOffset
  → updates SCNCamera position

New (SpriteKit):
  SKScene.update(_:currentTime)
  → reads HeadTracker.latestOffset  // same API!
  → maps to gaze point
  → hit-tests characters

HeadTracker needs zero changes.
Both SCNSceneRendererDelegate and SKScene.update
run once per frame. PokemonScene just reads the
same latestOffset / latestVelocity properties.

The only difference: SceneKit moves the camera,
SpriteKit maps to gaze coordinates.

View Swapping: SCNView ↔ SKView

Theme Switch Flow
Current architecture (simplified):
  DesktopWindowController
    └── contentView = SCNView (always)
          └── SceneViewController manages it

New architecture:
  DesktopWindowController
    └── contentView = SCNView  ← 3D themes (basketball, tree, sea...)
                   OR SKView   ← Pokemon theme

Switch sequence:
  1. User selects "Pokemon" from menu
  2. SceneViewController detects it's a SpriteKit theme
  3. Pause current SCNView rendering
  4. Create SKView (or reuse cached one)
  5. Set SKView.presentScene(pokemonScene)
  6. Swap window.contentView = skView
  7. Release old SCNScene to free memory
  8. Start SessionMonitor polling

Switch back to 3D theme:
  1. Stop SessionMonitor polling
  2. Swap window.contentView = scnView
  3. Load new 3D scene as usual
  4. Release SKScene + SKView

Both views share the same HeadTracker instance.
Window properties (level, click-through, etc) unchanged.

New Files Summary

PokemonScene.swift SKScene subclass. Grid layout, update loop (gaze + status), empty-state label. Entry point for the theme.
SessionMonitor.swift Timer-based scanner. Discovers sessions, parses JSONL, classifies activity, notifies PokemonScene via delegate/closure.
CharacterNode.swift SKNode subclass. Owns body sprite, status icon, label, hello bubble. Handles animations and state transitions.
CharacterGenerator.swift Pure function: sessionID → CGImage. Core Graphics drawing of procedural creatures.
Modified files SceneViewController.swift (add Pokemon entry, view swap logic), AppDelegate.swift (menu entry), DesktopWindowController.swift (support SKView as contentView)
